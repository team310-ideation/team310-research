# 데이터 소유 플랫폼 사례

## 한 줄 결론

미리(MIRI)는 “학생 데이터를 회사가 쌓아두는 입시 리포트 서비스”가 아니라 **학생이 자기 입시 데이터를 보관·이동·삭제·공유 권한까지 통제하는 학생 데이터 지갑**으로 설계해야 한다. 금융/의료 마이데이터와 Solid/1Password 사례를 보면, 핵심은 데이터 자체보다 `동의`, `전송`, `권한`, `이력`, `암호화`, `삭제` UX다.

## 참고 사례 비교

| 사례 | 핵심 구조 | 우리가 배울 점 | 바로 적용 가능성 |
|---|---|---|---|
| 금융 마이데이터 | 여러 금융기관의 개인신용정보를 API 방식으로 모아 조회·활용 | 학생부/활동기록도 “기관별 데이터 연결”로 생각해야 함 | 중간 |
| 건강정보 고속도로 | 개인 동의 기반으로 의료기관·공공기관 건강정보를 조회·저장·전송 | 학생 데이터도 본인 동의·전송 이력·기관 연결이 핵심 | 높음 |
| 개인정보 전송요구권 | 정보주체가 본인정보를 본인/다른 처리자/전문기관에 전송 요구 | 장기적으로 교육 데이터 포터빌리티 대응 필요 | 중간 |
| Solid Pod | 개인이 데이터 저장소를 갖고 앱이 권한을 받아 접근 | “학생 데이터 Vault” 설계의 개념 모델 | 중간 |
| 1Password | 사용자가 비밀정보를 암호화된 Vault에 저장 | 원본 학생부를 회사가 평문 장기 보관하지 않는 모델 | 높음 |

## 사례 1. 금융 마이데이터

금융 마이데이터는 사용자가 여러 금융기관에 흩어진 개인신용정보를 API 방식으로 모아 조회하고, 이를 기반으로 자산관리·추천 등을 받는 구조다.

### 미리에 주는 힌트

- 학생 데이터도 `학교`, `학생`, `학부모`, `학원`, `컨설턴트` 사이에 흩어져 있다.
- 초기에는 실제 기관 연동이 어렵기 때문에 `파일 업로드 기반 개인 데이터 지갑`으로 시작한다.
- 이후에는 학생이 자신의 데이터 패키지를 컨설턴트/학원/멘토에게 권한 부여하는 방식으로 확장한다.

### 적용 아이디어

```json
{
  "student_data_vault": {
    "owner": "student",
    "data_sources": ["student_record_pdf", "activity_log", "consulting_note"],
    "permissions": [
      {
        "grantee": "consultant",
        "scope": ["diagnosis", "roadmap"],
        "expires_at": "2026-06-30"
      }
    ],
    "audit_log": true
  }
}
```

## 사례 2. 건강정보 고속도로

보건복지부의 건강정보 고속도로는 개인 의료데이터 국가 중계플랫폼이다. 보건복지부 설명에 따르면 개인의 동의하에 여러 기관에 흩어진 의료데이터를 조회·저장하고 원하는 곳에 전송할 수 있다. 건강정보고속도로 포털은 FHIR 표준 기반 12개 항목의 건강데이터 전송·조회를 설명한다.

### 미리에 주는 힌트

- 학생 데이터도 `원본`, `요약`, `전송`, `활용 이력`을 나눠야 한다.
- 사용자가 “내가 누구에게 어떤 데이터를 보냈는지” 볼 수 있어야 신뢰가 생긴다.
- 단순 업로드가 아니라, `분석 후 삭제`, `내보내기`, `권한 회수`가 제품 핵심이 된다.

### 미리 버전으로 바꾸면

| 건강정보 고속도로 | 미리(MIRI) |
|---|---|
| 나의건강기록 앱 | 나의입시기록 Vault |
| 진료/투약/검사 기록 | 학생부/활동/상담/탐구 기록 |
| 의료기관 전송 | 컨설턴트/학원/멘토 공유 |
| 데이터 조회·저장·공유 이력 | 분석·다운로드·공유 이력 |

## 사례 3. 개인정보 전송요구권

개인정보보호위원회는 전 분야 마이데이터와 관련해 개인정보 전송요구권 제도 안내서를 공개했다. 개인정보보호법 개정 취지는 정보주체가 자신의 개인정보를 본인, 다른 개인정보처리자, 개인정보관리 전문기관 등에 전송 요구할 수 있도록 하는 방향이다.

### 미리에 주는 힌트

- 당장 법정 “개인정보관리 전문기관”이 될 필요는 없다.
- 하지만 서비스 철학은 `회사 소유 DB`가 아니라 `학생이 이동시킬 수 있는 데이터 패키지`여야 한다.
- 다운로드 가능한 JSON/PDF/Markdown 리포트가 필요하다.

## 사례 4. Solid Pod

Solid는 팀 버너스리 경의 웹 데이터 주권 구상에서 출발한 프로젝트다. Solid의 핵심 개념은 개인이 자기 데이터 저장소인 Pod를 갖고, 앱이 필요한 데이터에 권한을 받아 접근하는 구조다.

### 미리에 주는 힌트

- “미리가 모든 데이터를 소유한다”가 아니라 “학생 Vault에 미리가 접근한다”로 설계한다.
- 기능별 권한을 쪼갤 수 있다.
  - 학생부 진단만 허용
  - 면접 질문 생성만 허용
  - 컨설턴트 공유 7일 허용
  - 원본 파일 접근 금지, 요약본만 접근 허용

## 사례 5. 1Password

1Password는 사용자가 민감정보를 Vault에 저장하고, 암호화와 보안 모델을 통해 운영자가 쉽게 내용을 볼 수 없는 방향을 강조한다. 미리도 학생부 원본을 장기 평문 저장하는 방식은 피해야 한다.

### 미리에 주는 힌트

- 원본 학생부는 분석 후 즉시 삭제가 기본값이어야 한다.
- 장기 저장은 사용자가 명시적으로 켜는 옵션으로 둔다.
- 내부 운영자도 원본을 볼 수 없게 최소권한과 접근로그를 둔다.

## 미리(MIRI)의 데이터 소유 설계안

### MVP 단계

1. 사용자가 생기부/활동기록 파일 업로드
2. 서버에서 분석
3. 원본 파일 즉시 삭제
4. 분석 결과 JSON만 사용자 계정에 저장
5. 사용자가 Markdown/PDF/JSON으로 내보내기

### 6개월 단계

1. 학생별 Data Vault 생성
2. 분석 결과 버전 관리
3. 컨설턴트 공유 링크 생성
4. 공유 만료일/권한 범위 설정
5. 접근 이력 표시

### 장기 단계

1. 학교/학원/컨설턴트 데이터 연결
2. 학생의 데이터 전송요구권 대응
3. 교육 분야 마이데이터 또는 개인정보관리 전문기관 제도 변화 추적

## 제품 요구사항

| 요구사항 | MVP 필요 여부 | 설명 |
|---|---|---|
| 원본 파일 삭제 | 필수 | 분석 후 즉시 삭제 기본값 |
| 결과 내보내기 | 필수 | 학생이 자기 데이터를 가져갈 수 있어야 함 |
| 접근 로그 | 필수 | 누가 언제 봤는지 기록 |
| 권한 공유 | 6개월 내 | 컨설턴트/학원 공유 링크 |
| 암호화 저장 | 필수 | 저장 데이터 암호화 |
| 데이터 이동 API | 장기 | 제도/파트너 생긴 뒤 |

## 멘토링 반영: 소유할 수 있는 플랫폼

- 미리는 단순 컨설팅 리포트가 아니라 학생이 자기 입시 데이터를 소유할 수 있는 플랫폼이어야 한다.
- 공개된 범용 AI와의 차별점은 학생 데이터 누적, 아카이빙, 권한 공유, 삭제 가능성이다.
- 8개월 이상 이어지는 서비스 가치는 리포트를 한 번 생성하는 것이 아니라 학생의 기록과 전략을 누적하는 데서 생긴다.
- AI 사고위탁 자체는 장점이지만, 사용자에게는 "내 기록이 남고 다음 전략으로 이어진다"는 경험이 더 중요하다.

## 멘토에게 물어볼 것

- ❓ “학생 데이터 Vault”가 학부모에게 신뢰 요소가 되는가, 아니면 초기에는 너무 복잡한 메시지인가?
- ❓ 원본 파일을 저장하지 않는 구조가 컨설턴트/B2B 고객에게도 충분한가?
- ❓ 장기적으로 교육 데이터 마이데이터 포지션을 가져갈 수 있는가?
- ❓ "소유할 수 있는 플랫폼" 메시지가 초기 학부모에게 신뢰 요소가 되는가?
